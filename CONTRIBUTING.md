# Contributing

Thanks for contributing.

## Ground rules

- Keep the repository practical.
- Prefer concrete operating guidance over generic inspiration.
- Write in clear English.
- Keep examples small and easy to adapt.
- Avoid vendor lock-in where possible.
- Do not commit secrets, tokens, internal endpoints, or proprietary source code.

## Preferred contribution types

- clearer setup instructions
- better decision matrices
- improved prompt templates
- practical scripts that reduce repetitive work
- workflow variants for different toolchains
- notes on failure modes and recovery patterns

## Style

- Markdown first
- short paragraphs
- bullet lists only when they improve scanning
- commands must be copyable
- clearly label anything that is an example rather than a guaranteed command

## Pull requests

Please:

1. explain the problem being solved
2. describe the impact on the workflow
3. call out any assumptions
4. mention whether the change is tool-specific or generally applicable
