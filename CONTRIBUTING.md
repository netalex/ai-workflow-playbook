# Contributing

Thanks for contributing.

## Ground rules

- keep the repository practical
- prefer operating guidance over vague inspiration
- write in clear English
- keep examples easy to adapt
- avoid vendor lock-in where possible
- do not commit secrets, tokens, private URLs, or proprietary source code

## Preferred contribution types

- clearer setup instructions
- better script comments
- safer hook behavior
- better repomix configuration patterns
- stronger chunking or ingestion guidance
- examples of durable skills
- token-reduction improvements

## Style

- GitHub-flavored Markdown
- short paragraphs
- tables only when they improve scanning
- commands must be copyable
- examples must be labeled as examples

## Commit style

Use Conventional Commits and keep the summary meaningful.

See [templates/commit-style.md](templates/commit-style.md).

## Pull requests

Please:

1. explain the problem being solved
2. describe the impact on the workflow
3. call out assumptions
4. mention whether the change is tool-specific or generally applicable
5. mention any effect on token cost, security boundary, or durable knowledge
